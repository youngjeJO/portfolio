# 경력기술서 기반 포트폴리오 프로젝트 추천

> 본 문서는 프로젝트 내 경력기술서·About·프로젝트 상세(`src/content/about.ts`, `src/data/projects.ts`, `src/content/skills.ts`) 내용을 바탕으로, **간단한 것부터 복잡한 것까지** 포트폴리오로 만들 만한 주제 5가지를 정리한 것입니다.

---

## 경력 요약 (반영 기준)

- **역할**: Frontend-oriented BFF 웹 개발자 (경력 3년)
- **핵심 역량**: BFF 구조 설계, API Aggregation, 데이터 정규화, 실시간 데이터 처리, 상태 관리 아키텍처
- **기술 스택**: React 18, TypeScript, Zustand, Styled-components, SCSS, Axios, ASP.NET Core, OAuth 2.0, 쿠키 기반 세션, Cafe24/이니시스 API 연동
- **대표 성과**: 초기 로딩 약 30% 개선, API 호출 약 60% 감소, 바코드 스캔 기반 출고·검수 시스템, BFF 도입으로 프론트엔드 복잡도 감소

## 1. 간단한 프로젝트: Zustand + 쿠키 기반 설정 패널

**난이도**: ⭐  
**예상 기간**: 1~2일  
**경력 반영 포인트**: DHL ILMS의 “환경설정 시스템”(쿠키 기반 설정 저장, 유형별 커스터마이징), Zustand 상태 관리

### 개요
- 테마(라이트/다크), 테이블 페이지 크기, 알림 on/off 등 **설정값을 Zustand로 관리**하고, **쿠키에 영구 저장**하는 소규모 설정 UI.
- 재방문 시 쿠키에서 복원해 동일한 설정으로 복원되는 것까지 구현.

### 구현 포인트
- `zustand` + `persist` 미들웨어 또는 직접 쿠키 read/write.
- 설정 변경 시 즉시 쿠키 반영 및 상태 동기화.
- Styled-components 또는 SCSS로 간단한 폼/토글 UI.

### 포트폴리오 메시지
- “물류 시스템에서 사용한 **쿠키 기반 설정 저장**과 **Zustand 상태 관리**를 소규모 데모로 재현.”

## 2. 간단한 프로젝트: 다중 공개 API 집계(API Aggregation) 데모

**난이도**: ⭐⭐  
**예상 기간**: 2~3일  
**경력 반영 포인트**: Cafe24/GODO BFF의 “여러 API를 단일 엔드포인트로 집계”, 프론트엔드 요구사항 기준 응답 설계

### 개요
- **프론트엔드 1개 화면**을 위해 필요한 데이터를 “여러 공개 API”에서 가져와 **한 번에 묶어서 보여주는** 미니 BFF 데모.
- 예: 날씨 API + 뉴스 API + 지도 API를 하나의 “대시보드용 API”로 묶고, React에서 **한 번의 호출**로 받아서 표시.

### 구현 포인트
- Node.js(Express) 또는 ASP.NET Core로 **단일 엔드포인트** 구현.
- 내부에서 여러 외부 API 호출 후 **화면에 맞는 형태로 재구성(정규화)** 해서 반환.
- React + Axios로 해당 엔드포인트만 호출하고, 로딩/에러 처리.

### 포트폴리오 메시지
- “실무에서 적용한 **BFF·API Aggregation** 개념을 공개 API로 검증 가능한 형태로 구현.”

---

## 3. 중간 난이도: 병렬 API 호출 vs 순차 호출 비교 데모

**난이도**: ⭐⭐⭐  
**예상 기간**: 3~5일  
**경력 반영 포인트**: Cafe24 BFF의 “병렬 API 호출(Promise.all / allSettled)”, “초기 로딩 약 30% 개선, API 호출 약 60% 감소”

### 개요
- **동일한 데이터**를 “순차 호출” vs “병렬 호출”로 가져오는 두 가지 방식을 구현하고, **로딩 시간·호출 횟수**를 시각적으로 비교하는 페이지.
- Mock API(예: JSONPlaceholder 여러 리소스) 또는 자체 Mock 서버 사용.

### 구현 포인트
- 순차: `async/await`로 API를 하나씩 호출.
- 병렬: `Promise.all` 또는 `Promise.allSettled`로 한 번에 호출.
- 각 방식별 **소요 시간 측정** 후 UI에 표시(예: “순차 1.2s vs 병렬 0.4s”).
- (선택) “Before/After” 스타일로 경력기술서의 **30% 개선, 60% API 감소** 스토리를 짧은 문구로 설명.

### 포트폴리오 메시지
- “실서비스에서 적용한 **성능 최적화(병렬 호출·불필요 호출 제거)** 를 재현 가능한 데모로 정리.”

## 4. 중간 난이도: OAuth 2.0 + 쿠키 세션 플로우 데모

**난이도**: ⭐⭐⭐  
**예상 기간**: 4~6일  
**경력 반영 포인트**: Cafe24 BFF의 “Cafe24 OAuth 2.0 인증”, “쿠키 기반 세션 유지 및 토큰 자동 갱신”, GODO의 “인증·세션·에러 처리 공통화”

### 개요
- **GitHub / Google OAuth** 등 공개 OAuth를 사용해 “로그인 → 쿠키에 세션 저장 → 토큰 갱신(또는 만료 처리)”까지를 **BFF + React** 구조로 구현한 데모.
- 실제 결제/운송장이 아닌 **인증·세션 플로우**에만 집중.

### 구현 포인트
- BFF(Express 또는 ASP.NET Core)에서 OAuth redirect·callback 처리, 쿠키 설정.
- 프론트엔드는 BFF의 “로그인 여부 확인” API만 호출, 쿠키는 httpOnly 등으로 보안 고려.
- (선택) 토큰 만료 시 갱신 또는 “세션 만료” 모달 후 로그인 페이지 리다이렉트.
- 인증 실패/만료 시 **공통 에러 처리** 한 곳에서 처리하는 구조로 설계.

### 포트폴리오 메시지
- “Cafe24/GODO에서 사용한 **OAuth + 쿠키 세션 + 공통 인증 처리**를 공개 OAuth로 재현.”

## 5. 복잡한 프로젝트: 미니 “주문·배송 통합 화면” + BFF 전체 구조

**난이도**: ⭐⭐⭐⭐  
**예상 기간**: 1~2주  
**경력 반영 포인트**: Cafe24 운송장/GODO 이커머스의 “주문·결제·배송을 하나의 화면·API 흐름으로 단순화”, “BFF 설계 + React + 상태 관리 + 데이터 정규화” 전반

### 개요
- **한 화면**에서 “주문 목록 + 결제 상태 + 배송 상태”를 함께 보는 **운영자용 미니 대시보드**.
- 백엔드는 **Mock 서버**(주문 API, 결제 API, 배송 API를 각각 분리)로 구성하고, **BFF 레이어**가 이들을 집계·정규화해 **프론트엔드 전용 API 1~2개**로 제공.
- React + TypeScript + Zustand로 해당 API만 소비하고, 로딩/에러/빈 상태 처리.

### 구현 포인트
- **Mock 백엔드**: 주문/결제/배송을 시뮬레이션하는 3개의 간단한 API(Express 또는 ASP.NET Core).
- **BFF**: 화면 단위로 “주문+결제+배송”을 한 번에 묶는 엔드포인트, 응답을 UI용 구조로 정규화.
- **프론트엔드**: Zustand로 대시보드 상태(목록, 필터, 로딩/에러), useShallow 등으로 불필요 리렌더 최소화.
- (선택) “이니시스 결제 흐름”은 **시뮬레이션**(팝업 + PostMessage로 결제 완료/실패만 처리)으로 대체 가능.

### 포트폴리오 메시지
- “물류·이커머스 실무에서 설계한 **BFF + API Aggregation + 데이터 정규화 + React 상태 관리**를 하나의 데모로 통합해 보여줌.”
